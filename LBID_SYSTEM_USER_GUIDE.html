LBID System Guide

A professional system analyst and business analyst view of LBID: product mechanisms, user workflows, page responsibilities, backend APIs, database architecture and launch readiness.

Production: https://lbid.vercel.app

1. Product Overview

LBID is a sealed-bidding logistics marketplace connecting overseas agencies with Hong Kong forwarders. It replaces fragmented email, phone and manual quotation workflows with a structured digital transaction process.

Core objectSR

Shipment Request as the demand record.

Core transactionBid

Forwarder submits sealed quote with token deduction.

Core outputOrder

Accepted bid creates order workspace.

Positioning: LBID is a workflow platform, not the carrier of record. It supports matching, quotation, documents, communication, tracking and audit.

2. Business Mechanisms

Dual-capability company account

One company can enable Client capability, Forwarder capability, or both. This reflects real logistics business behavior.

Admin-reviewed SR

Shipment requests enter review before publication, protecting marketplace quality and data clarity.

Sealed bidding

Forwarders submit confidential bids. Competitor prices and identities remain hidden during the bid window.

Hybrid award model

The lowest valid bid is highlighted, but the client can choose another bid based on value, trust or service fit.

Token transaction

Bid submission deducts tokens through Supabase RPC to protect the token ledger.

Order workspace

Awarded bids become operational workspaces with documents, messages and tracking.

6. Database And RPC

Main tables

users, company_profiles, shipment_requests, bids, quotations, orders, documents, messages, tracking_events, notifications, audit_logs.

Critical RPC

submit_bid_with_token, adjust_token_balance, accept_bid_to_order. These protect commercial transaction integrity.

7. System Architecture

Browser→Next.js on Vercel→API Route Handlers→Supabase Auth→PostgreSQL + RPC→Storage / Realtime / Email / Payment
Frontend

Next.js App Router, SiteShell, SessionGate, live workspace components.

Backend

Next.js route handlers, Supabase session verification, service role for controlled writes.

Infrastructure

Vercel deployment, Supabase database/auth/storage, Resend email, Stripe/FPS payment.

8. Security And Governance

  • Protected pages redirect unauthenticated users to login.
  • API routes require Supabase access token.
  • Service role key must never be exposed to frontend.
  • Token deduction must remain RPC/transaction based.
  • Admin, payment, bid acceptance and order status changes should be audit logged.
  • Order data should only be accessible by order parties or admin.

9. User Operation Guide

Client

Login → onboarding → create SR → wait for admin approval → compare bids → accept bid → manage order.

Forwarder

Login → set routes/profile → view marketplace → submit sealed bid → wait for result → manage awarded order.

Admin

Review SR → approve/reject → verify accounts → confirm payment → monitor audit log.
